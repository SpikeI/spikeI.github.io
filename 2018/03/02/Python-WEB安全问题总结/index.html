eval的不正确使用

有些人会使用eval当作json decode来用，这其中就可能引发安全问题。当封装eval的时候将__builtins__置空，不能达到完全安全的效果。

可以通过().__class__.__bases__[0].__subclasses__()找到基类Object，然后在寻找他的子类进行调用。

同时如果知道一个模块中调用了os模块，我们可以通过使用zipimporter来进行加载后调用。

应该也算是沙箱逃逸了。

不安全的解压引发的代码执行

​ 其实这个问题很好理解，就是再解压的时候将文件解压到自动加载的文件中去，flask中的每个模块下都会有一个__init__.py，当加载这个模块的时候就会自动加载里面的内容作为初始化的代码。

​ 想象一个这样的场景，一个解压的程序。没有做好严格的业务功能审查，导致可以解压到任意文件夹下（解压路径没有进行限制）。那么我们就可以将__init__.py解压到flask的一个模块下，从而实现代码执行。

原文：https://ajinabraham.com/blog/exploiting-insecure-file-extraction-in-python-for-code-execution

Django安全

  • CVE-2017-12794 debug page XSS。在注册两个相同用户名的时候，debug的返回页面中有XSS。漏洞版本 1.11.4及以下。1.11.5中增加一个增强转义来修补这一漏洞。
  • Django安全机制：
    • xss中<、>、&都会进行实体转义，属性中的动态内容记得加双引号"。
    • settings.py中可以开启csrf，这样在每个表单中都可以通过csrf_token字段来加入csrf。其中装饰器@csrf_exempt是去除csrf防护，并且在Django中默认Get方法也不会有csrf校验。
    • Django在正常的sql语句中，都会对参数进行过滤。有两个例外，在extra的where的参数中就不会有转义。在直接拼接时是没有转义的。在有params的时候是有转义的。
    • 自带防止点击劫持（禁止iframe、frame中使用）的功能X-Frame-Options middleware。
    • Django中django.http.HttpRequest.get_host()可以获取host头，有伪造的话会报错。

参考http://xxlegend.com/2015/04/01/Django%E5%AE%89%E5%85%A8%E6%9C%BA%E5%88%B6/

  • 16年有个bypass csrf的洞，其原理是只要post字段的csrfmiddlewaretoken和cookie中的cstf字段中的值相等，那么就会过csrf的校验。此时我们只需要控制cookie即可。在python版本小于 2.7.9和小于3.2.7版本中的cookie.py遇到超出预期的字符时会将其分割成两个键值对。即会将a=1]b=2解析成a=1和b=2。如此我们只需要控制一个cookie字段，将其设置成xxx]csrf=wewant即可。然而使用了google analytics的网站google analytics会自动加入一些cookie。其中会将referer加入到一个字段中，所以我们只需要讲referer为http://x.com]csrf=wewant即可出发漏洞。

参考https://paper.seebug.org/58/

  • CVE-2017-7233 bypass django 的 is_safe_url()。用十进制数字代替ip地址就会绕过安全性检验。

    CVE-2017-7234 url跳转漏洞。在访问/staticp/*目录下的文件时会直接访问static目录下的文件，在这过程中有个逻辑的bug，会对uri进行进一步加工，如果加工结果和原来的结果不同。那么就直接跳转到新的uri中。poc:/staticp/%5c%5cABC.com

参考：https://paper.seebug.org/274/

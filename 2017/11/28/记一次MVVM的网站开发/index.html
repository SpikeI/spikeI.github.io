记一次MVVM的网站开发

网站开发有了先前的经验，相对有着手点，但其中包含更多的问题在其中。

  1. 功能需求
  2. 新型框架的使用。VUE、Flask
  3. 任务的分配，代码的维护
  4. 稳定性分析

功能需求

​ 功能需求调查的不是很明确，使得开发效率很低。甚至最后的实用性都不是很强。以此为戒（虽然这次是抱着学习网站开发技术的心态来做的，XD）。

任务的分配、代码的维护

​ 分成两个组分别进行开发，每个组维护一条dev线。master融合其他线。每组中分为前端和后端。维护git的代码的确很浪费精力。分成两个分支后，若分支分离的不是特别干净，在每次融合到master的时候都需要进行处理冲突。每个晓得模块实现完成后都要和master进行融合并且打tag。

​ 由于人手有限，时间紧张。前端后端的接口详情通过维护一个文本文件的方式进行沟通，首先根据需求制定好接口信息，将其写入文件中，前端和后端再根据这个接口进行开发。

​ 在开发前，先定好变量，注释等规范。尽量丰富代码的规范性，在合理的范围内尽量对代码进行封装，增加可读性，并且可以及时溯源。

VUE和Flask

​ 通过一段时间对VUE、flask的学习，才开始进行开发。

​ flask简单优雅，蓝图功能更使得水平扩展变得简单。很多wtform、message、login等插件很好用。但这里我们用flask作为api接口，界面和路由全部由VUE实现。

​ VUE实现路由和界面，参考了饿了么和pwnhub。使用element ui中的插件。

​ 其中产生了一个问题就是restful uri在使用过程中如果从本页面跳转则可以正常访问，若直接访问则无法正常访问。解决办法有两种一是讲所有的uri跳转都跳到我们的入口页面上。而是将404设置成路由页面。

​ 还有一些细节，比如权限的控制和后台地址的发放，不同级别的用户得到的信息是不同的，可获取用户数据的api是经过严格访问控制的。cookie使用httponly的app.config["SESSION_COOKIE_HTTPONLY"] = True，使用CSRF token来确保不被CSRF攻击。上传文件的文件名有特殊要求，并且对上传文件的文件名进行特殊处理，使上传者不能预测到文件名，且用户名都将以合法字符保存。由于在vue中自动进行了实体编码转换，所以暂时没有特殊考虑XSS的问题。关注了Flask模板自身的漏洞，模板注入（该漏洞需要flask和jinja2配合使用，我们没有该问题。）

​ 在网页布局和前端方面耽误了很长时间，在那之后对前端大大的敬畏之心又加深了不少。

其他工作

​ 最后，开发之前要考虑维护成本以及维护人员的数量。依此来进行一些开发任务的选择，舍弃一些维护成本比较高的，以实用性稍差点，但维护成本比较低的来代替。
